# omcp v1.7 → v2.0 roadmap

**Status**: living document
**Created**: 2026-05-25 (post v1.6.0 cut)
**Owner**: omcp maintainers

This is the **single source of truth** for the v1.7 → v2.0 plan. Previous
versions had forward-looking content split between `HANDOFF.md` ("vN
follow-ups") and `CHANGELOG.md` ("What's still pending") sections; those
accumulated drift across releases. This doc consolidates the forward
plan; HANDOFF and CHANGELOG remain retrospective (per-version deliverable
snapshots).

---

## Operating principles (user-mandated, hard)

1. **不准问题往后迁移** — no kicking the can. Within-version findings get
   solved in-version; cross-version carry-forward needs explicit ADR
   justification, not "we'll get to it later".
2. **多 agent 不同方向直到解决** — single-agent stuck for >30 min →
   spawn 3+ subagents with different attack angles in parallel. Stop
   only when one direction works OR all directions exhausted with
   documented ADR.
3. **用事实说话** — deterministic vitest is primary evidence; smoke is
   secondary sanity check.
4. **每个 fix 后 team+critic** — architect + critic independent context
   review per fix.
5. **多个 subagent 独立上下文验证** — cross-verification across
   independent contexts.
6. **读上游文档再下结论** — exhaust upstream sources before blaming
   upstream.
7. **omc-style commit trailers** — Constraint / Rejected / Confidence
   / Scope-risk.
8. **9 invariants 不许违反** — assertSafeSlug / atomicWriteFileSync /
   4-manifest sync / valid events / subagentStart casing / escapeRegExp
   / banned tokens / CLI registration / pidfile+stop-verb.
9. **每个 task 一个 commit**.
10. **Tag gate = live-evidence-based** — every tag requires a live
    smoke artifact proving the user-facing problem is solved.

## Architecture rationale (post-v1.6)

omcp at v1.6 has 42 skills and 19 agents (skill surface ahead of omc's
32). The remaining gap to "omc-class product" is NOT skill count — it's:
- runtime maturity (each mode actually works end-to-end against real
  Copilot, not just deterministic mocks)
- MCP server integration verified live
- packaging + release pipeline (npm publish, marketplace)
- user-facing docs

These four are the spine of v1.7 → v2.0.

"MCP server X removed in v1.8 because <reason>" — legitimate per
no-kicking-can principle).

### Tier 4 — 19 agent QA matrix

## v1.9 — Product polish + sparkshell (Windows-first ship target)

**Scope**: 2 sessions. **Theme**: Windows-only experience made cohesive
+ sparkshell built per user "两条腿走路" decision.

### Tier 1 — HUD completion

- column 1+2 + 6 wire (currently column 3-5 done per v1.3.0)
- multi-mode active display
- `omcp hud` as Copilot statusLine.command, refresh cadence acceptable

### Tier 2 — Error messages + onboarding

- All fail paths produce actionable errors (v1.5 EACCES is the pattern;
  apply systematically across the codebase)
- `omcp doctor` gains 5+ new checks: mcp-config integrity, settings.json
  drift detection, hook registration consistency, agent catalog
  completeness, plugin install state
- `omcp doctor --json` machine-readable output
- `omcp setup --interactive`: first-install wizard for Copilot auth +
  model family + initial doctor run

### Tier 3 — sparkshell .exe wrapper (USER-MANDATED, 两条腿走路)

Per 2026-05-25 user decision: build sparkshell **regardless** of whether
upstream fixes the pwsh dispatch bug. Outer-loop (v1.6) stays as primary
ralph path; sparkshell rescues Stop-only features (L1.3 compaction
advise, per-tool-call hook delivery, HUD per-event refresh).

**Steps**:
1. Install Rust toolchain on the dev machine
2. Bootstrap `crates/sparkshell` (Cargo workspace add)
3. Write minimal Rust .exe: argv[1] = event name; stdin = JSON
   payload; spawn `node` + abs path of `dist/cli/omcp.js` + `hook fire
   <event> --json`; pipe stdout/stderr/exit through
4. 4 D-tests (cargo test) + 1 live smoke against real Copilot
5. **Multi-direction team if Rust direction stalls**: (A) Rust .exe;
   (B) pure .cmd batch + `cmd /c` wrapper; (C) Node `child_process.fork`
   IPC. Any direction working → ship.
6. Three directions all fail → ADR "Windows hook dispatch
   unsolvable from omcp-side per current Copilot 1.0.53 dispatch
   semantics", v1.9 other content still ships.

### Tier 4 — Misc polish

- Stream/incremental log scan in `probeHookDeliveryHealth` (v1.5 minor)
- `omcp state timeline <session-id>` for single-run observability
- ralph stderr prints `[iter N/max] now executing...` between spawns

### v1.9 tag gate (hard)

1. HUD all 6 columns wire
2. `omcp doctor` has 5+ new checks all pass on clean install
3. sparkshell works live OR upstream is fixed live OR ADR documents
   why both fail
4. Onboarding wizard completes a fresh install → first ralph run

## v2.0 — Release (Windows-first per user 2026-05-25)

**Scope**: 2 sessions. **Theme**: ship to users.

### Tier 1 — npm publish

- `oh-my-copilot` v2.0.0-rc.1 → npm
- Fresh-machine install test: `npm install -g oh-my-copilot` + `omcp
  setup` from scratch on a clean Windows dev box
- `npm pack --dry-run` size audit; trim unnecessary files via `files`
  array refinement
- Bin entries + postinstall script validated

### Tier 2 — Copilot marketplace listing

- Submit per Copilot plugin marketplace requirements
- Marketplace approval is **NOT** a blocking gate for npm release;
  npm ships first, marketplace approval can land later

### Tier 3 — Docs site

- mkdocs-material or docusaurus
- Required pages:
  - Install + quickstart
  - One tutorial per LOOPING_MODE (6)
  - Reference per MCP server (11)
  - Troubleshooting (incl. upstream pwsh dispatch known-bug)
  - Architecture (link to invariants.md, this roadmap)
  - Migration v1.x → v2.0
- 5-min screencast demo: install → omcp setup → omcp ralph on toy PRD →
  output

### Tier 4 — Stability commitments

- Support matrix: Copilot 1.0.53+ / Node 20+ / **Windows 11
  (primary)**
- **Explicitly state**: v2.0 is Windows-first; cross-platform v2.1+
  (user 2026-05-25 decision; not a deferral, an explicit scope choice)
- API stability: v2.0 ralph/autopilot/team CLI shape is stable;
  breaking changes require v3.0

### v2.0 tag gate (hard)

1. rc.1 completes fresh-machine install + setup + first ralph run
2. Docs site live (URL routable)
3. At least 3 external users (not the dev) have run omcp and given
   feedback
4. v2.0 = "Windows-first" explicitly stated in README, CHANGELOG, docs
