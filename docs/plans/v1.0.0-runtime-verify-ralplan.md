### Scenario 1 — Smoke (Phase A) fails at hook dispatch with `SyntaxError: Unexpected token ':'` (the OMC-style symptom)

**Likely cause:** OMC plugin not actually disabled (settings.json write failed silently), so OMC's broken `$CLAUDE_PLUGIN_ROOT` hooks still fire and pollute `process-*.log`. The omcp hook fires CLEAN alongside, but the visible noise looks like omcp is broken.

### Scenario 2 — Smoke (Phase A) succeeds initially but iteration counter never advances past 1

**Likely cause:** `incrementRalphIteration` reads `ralph-state.json` from `getOmcpRoot(worktreeRoot)` — if the hook fires in a different working directory than ralph started in, the state file is invisible. Or: `Stop` event doesn't fire at all (Copilot version mismatch) so the increment never runs.

**Detection:** Mid-smoke, `cat .omcp/state/ralph-state.json` after the first Copilot turn — should show `iteration: 2`. If still 1, `grep "agentStop\|Stop" ~/.copilot/logs/process-*.log` — if no Stop event entries, the hook itself never ran.

**Mitigation:**
- Smoke artifact captures the ralph-state.json snapshot at start, mid-run (after first Copilot turn), and end.
- If iteration stays at 1: the `worktreeRoot` parameter resolution in the hook is suspect — instrument with a debug log line.
- If Stop event never fires: Copilot version regression — re-probe with `scripts/smoke/wire-probe-for-tui.mjs`.

### Scenario 3 — Item B fallback implementation breaks an existing test

**Likely cause:** `omcpHookCommand()` is called from multiple paths (setup.ts, doctor.ts, possibly future use). Adding PATH-detection logic with synchronous `which` may inject a process exec into a code path that was pure. Tests that mock `mergeCopilotHooks` may fail if the new behavior bypasses the mock.

**Detection:** Run `npx vitest run src/__tests__/copilot-config*.test.ts src/__tests__/copilot-config.hook-target.test.ts src/cli/commands/__tests__/setup.test.ts` after each edit. Watch for new failures.

**Mitigation:**
- Implement PATH detection as a **pure** check (e.g., scan `process.env.PATH` split + `fs.existsSync` on candidate paths) — NOT a child_process spawn. This keeps the function pure-sync and mockable.
- The `omcpBin` option in `MergeHookOptions` already exists as an override; tests pass an explicit `omcpBin` to avoid hitting the new detection. Document this in the JSDoc.

### Phase C — `omcp verify-phase <phase-id>` CLI implementation

**Naming rationale (Architect iter-1):** the bare `omcp verify <task>` slot is already claimed at `src/cli/omcp.ts:85` by the `"verify"` entry in `MODE_COMMANDS`, which routes to `skills/verify/SKILL.md` (a generic verification skill). Adding a second `.command("verify ...")` would shadow or collide depending on Commander version. The new verb is therefore `omcp verify-phase <phase-id>` — distinct slug, no collision, and explicit about its team+critic-protocol semantics.

**Scope:**
1. **New module:** `src/cli/commands/verify-phase.ts` exporting `runVerifyPhase({ phaseId, maxIterations?, spawn? })`.
   - Inputs: phase-id (validated via `assertSafeSlug`), optional maxIterations (default 5), optional injectable `spawn` function for testability (DI pattern matching `src/cli/commands/exec.ts:28-32`).
   - Reads `.omcp/state/verification/<phase-id>-submission.md` (per protocol Step 1). If absent → exit 2 with clear error.
   - Spawns architect: `copilot -p` with the protocol's architect-review prompt template, captures stdout → parses verdict via the new `detectVerdict()` helper that distinguishes APPROVE / ITERATE / REJECT (the existing `detectArchitectApproval` in `src/lib/ralph-state.ts:438` matches the `<architect-approved>VERIFIED_COMPLETE</architect-approved>` sentinel; `detectVerdict` is the superset that also recognizes ITERATE/REJECT keywords).
   - Spawns critic (independent session id): same shape, separate prompt.
   - Applies Step 4 pass condition: both APPROVE → exit 0; either REJECT → exit 1 with escalation file written; otherwise loop (Step 5) with iteration counter incremented and submission re-read each cycle.
   - Writes `.omcp/state/verification/<phase-id>-<run-id>.json` via `atomicWriteFileSync` (assertSafeSlug on the run-id).
   - Exit codes: 0 (pass), 1 (escalation), 2 (invocation error).

2. **New helper in `src/lib/ralph-state.ts` (or a new `verdict-parser.ts`):** `detectVerdict(text): 'APPROVE' | 'ITERATE' | 'REJECT' | null`. **Strictness rule (per Critic iter-1):** the verdict keyword must appear as the sole significant token on a line — leading/trailing whitespace and optional markdown bold (`**APPROVE**`) tolerated, but inline occurrences like `"I would APPROVE this if X"` or `"REJECT the alternative"` must NOT match. Case-insensitive. If a line contains more than one verdict keyword, return null (ambiguous). Cross-references the existing `detectArchitectApproval` (already approves on `<architect-approved>VERIFIED_COMPLETE</architect-approved>`) so the two surfaces stay consistent. A future post-v1.0.0 enhancement may move to a full sentinel protocol (`<verdict>APPROVE</verdict>`) — out of scope for v1.0.0.

3. **Register CLI:** `src/cli/omcp.ts` — `.command("verify-phase <phase-id>").option("--max-iterations <n>", ..., 5)` calling `runVerifyPhase`. The slug `verify-phase` is chosen specifically to avoid collision with the existing `"verify"` entry in `MODE_COMMANDS` at `src/cli/omcp.ts:85` (which routes to `skills/verify/SKILL.md`).

## Architecture Decision Record (ADR)

**Decision:** Verify orchestrator-v1 by driving a real Copilot CLI end-to-end smoke test first; then close the PATH-fallback pre-mortem; then ship the `omcp verify` CLI verb; then cut v1.0.0. Defer team-shard real smoke and daemon mode to post-v1.0.0.

**Drivers:**
1. orchestrator-v1's code-completeness is unproven — the HANDOFF explicitly distinguishes "code-complete" from "verified-working" and stakes v1.0.0 on the latter.
2. Pre-mortem #1 (omcp on PATH) was promised in the prior plan but never implemented; v1.0.0 must close it.
3. The team+critic protocol exists as a doc; making `omcp verify` real automates a workflow the user already manually invokes.

**Alternatives considered:**
- **V-B** (pre-mortem closure before smoke): rejected — inverts user priority and the fallback isn't a smoke gate when we bootstrap via `npm link`.
- **V-C** (skip smoke, cut v1.0.0): rejected — would regress project discipline; HANDOFF explicitly forbids this.
- **V-D** (smoke + all pre-mortems + speculative additions): rejected — scope creep; orchestrator-v1's plan defers daemon and modifiedArgs.
- **BS-2** (`node dist/cli/omcp.js setup` instead of npm-link): rejected — conflates Phase A and Phase B failure modes.
- **BS-3** (implement B first, then A): rejected — user explicitly orders A before B; the bootstrap via npm link is cheap enough.

**Why chosen:**
- V-A + BS-1 matches user directive verbatim.
- Smoke artifact is the most direct evidence orchestrator-v1 is real.
- Fallback test landing in Phase B independently validates pre-mortem #1.
- `omcp verify` CLI completes the team+critic protocol's tool surface.
- Small tails (E1, E2) close two known Critic iter-2 carry-overs in commits scoped tight to their concerns.

**Consequences:**
- One `npm link` side-effect at session start; one `npm unlink` (or persist) at end.
- OMC plugin disable+re-enable bracket around smoke run — must be precise so user's normal state is restored.
- `omcp verify` adds 1 CLI verb + ~200 LOC + 10+ tests; minimal blast radius.
- v1.0.0 manifest bumps touch 4 files atomically (invariant 3 enforced).
- The team-shard real smoke and daemon mode (orchestrator-v1's deferred items) remain deferred to a later release.

**Follow-ups (post-v1.0.0):**
- D (OMC upstream patch for `$CLAUDE_PLUGIN_ROOT`) — separate session, separate repo.
- Team-shard real smoke (orchestrator-v1 plan Phase 3.T3, never end-to-end verified).
- Daemon mode (Option O-B of orchestrator-v1, deferred).
- modifiedArgs surgeon mode (Phase 7, gated on TUI smoke PASS — now we have that gate).

A clean-environment job that runs `npm pack && npm install -g ./oh-my-copilot-*.tgz && omcp setup && omcp doctor` would close the seam between Phase A's smoke (which uses `npm link`) and the real first-time-user experience (which uses `npm install -g`).
