# Ralplan angle C — risk-prioritized

**Baseline (verified 2026-05-25)**: HEAD = `v1.7.0 @ 8bfea68`; vitest 1178 passing / 2 skipped / 1 pre-existing worker-fork EPERM file-level baseline; tsc clean.
**Attack frame**: order by *unknown-risk*, not by roadmap aesthetics. Fail fast on whatever can most catastrophically blow up the v1.8 → v2.0 arc, especially items that cascade-block downstream versions (HUD, doctor, docs site, marketplace).
**Risk class legend**:
- **CATASTROPHIC** — failure cascades into v1.9 and/or v2.0, may force ADR-chains or default-list amputations. Highest info-gain to hit FIRST.
- **MAJOR** — failure is contained within v1.8 but eats 1+ session of multi-direction team work; non-trivial recovery.
- **MINOR** — failure is local, deterministic, fixable in <30 min; no downstream cascade.

The roadmap suggests "Phase 1 MCP first" (handoff §"Phase 1"). Angle C agrees — but tightens the ordering further: among MCP servers, hit the **three most likely to be broken** first, because if 3+ MCP servers are broken we learn something architectural (server-runtime regression vs. per-server bug), and that pivots the rest of v1.8 plus all of v1.9 doctor+HUD column 6.

## Per-story breakdown

### S-MCP-CANARY — E2E-verify 3 canary MCP servers FIRST

**Risk class**: CATASTROPHIC

**Why this risk**: 11 MCP servers have **never** been live-verified end-to-end. v1.5 confirmed they CAN start (deps install); we have NO evidence they respond to JSON-RPC from a live `copilot` session. If 3 servers fail in the same way → it's a `server-runtime.ts` bug (the shared wrapper at `src/mcp/server-runtime.ts:25` — `runMcpServer({ name, version, tools })`). If 3 servers fail in 3 different ways → it's the slow grind of per-server hand-fixing. The shape of the failure decides whether v1.8 fits in 1 session or 3.

**Which 3 servers, and why these 3** (chosen for maximum information gain about failure modes):
1. **`state-server.ts`** — smallest tool surface, most-used by mode.ts + cancel. If THIS fails, runtime wrapper is broken (high blast radius). Probe: `state.read` + `state.write` round-trip.
2. **`notepad-server.ts`** — touches `.omcp/notepad.md` (filesystem write). Tests Invariant 2 (atomicWriteFileSync) at the MCP boundary. Probe: `notepad.write_priority` + `notepad.read`.
3. **`code-intel-server.ts`** — most complex tool surface (LSP-style: hover/refs/rename/diagnostics). Tests user-input regex handling at `code-intel-server.ts:567-589` (Invariant 6 known-exception site). Probe: `code-intel.workspace_symbols` against a tiny test fixture project.

These 3 cover: (a) the simplest server (isolates runtime wrapper bugs), (b) a filesystem-mutating server (isolates atomic-write/invariant boundaries), (c) the most complex server (surfaces input-handling fragility). If runtime wrapper is the problem, ALL THREE fail identically — diagnostic gold.

**Acceptance criteria**:
- 3/3 deterministic harness vitests: spawn server entry → stdin JSON-RPC ping → assert valid response shape (id echoed, no `isError`, `tools` array non-empty for `tools/list`).
- 3/3 live e2e: from a real `omcp autopilot "<tiny task that needs the server>"` smoke, observe successful tool call + return.
- Smoke artifact written to `docs/smoke/v1.8-mcp-canary-{state,notepad,code-intel}.md`.

**What could fail**:
1. **`server-runtime.ts` regression** — runtime wrapper handler shape doesn't match current `@modelcontextprotocol/sdk` version. All 3 canaries fail identically. Recovery: pin SDK + audit `setRequestHandler` signature. ~2-4h.
2. **stdio framing broken on Windows** — `StdioServerTransport` newline/encoding mismatch with Copilot's MCP client. ALL 11 servers blocked. ~1-2 sessions of multi-direction team work.
3. **Per-server tool handler bugs** — 1-of-3 fails uniquely; treat as ordinary v1.8 work. ~1-2h per server.
4. **Copilot mcp-config.json schema drift** — Copilot 1.0.53 expects a config shape omcp doesn't emit. Diagnostic: server spawns but Copilot never calls. ~2-4h to align.
5. **`assertSafeSlug` rejecting MCP tool arg paths** (Invariant 1) — code-intel canary fails on workspace fixture path. ~1h.

### S-MCP-CHECKPOINT — Early-discovery checkpoint after canary

**Risk class**: CATASTROPHIC (it's the safety valve)

**Why this risk**: Without an explicit pause, the session burns through all 11 servers using a hypothesis that may be wrong. The checkpoint exists to **replan** before sinking more time.

**Acceptance criteria**:
- After S-MCP-CANARY completes, write a one-page diagnosis:
  - 3/3 pass → proceed to S-MCP-REST (8 remaining servers) with confidence.
  - 2/3 pass → proceed but with elevated per-server attention; pre-allocate 30 min/server.
  - 1/3 or 0/3 pass → STOP. Spawn 3-direction team on `server-runtime.ts` / stdio framing / Copilot mcp-config shape. Do NOT proceed to S-MCP-REST until shared cause isolated.
- Decision recorded in `docs/architecture/v1.7-to-v2.0-roadmap.md` v1.8 working section.

**What could fail**: The temptation to "just push through" on partial-pass results. Mitigation: hard-gate this checkpoint with explicit user-side approval before resuming S-MCP-REST when >=1 canary fails.

### S-MCP-REST — Remaining 8 MCP servers (e2e + deterministic harness)

**Risk class**: CATASTROPHIC (cumulative — each broken server cascades into v1.9 doctor + v2.0 docs)

**Servers**: `trace`, `project-memory`, `loop`, `hermes`, `wiki`, `python-repl`, `shared-memory`, plus the server-runtime entry's own ping test.

**Why this risk**: Per the roadmap "MCP server X removed in v1.8" precedent — explicit removal is allowed BUT each removal nukes a v2.0 docs reference page and a v1.9 doctor check. The accumulated removal cost is what makes this catastrophic in aggregate.

**Acceptance criteria**:
- 11 vitests (one per server) at `src/__tests__/mcp-server-*.test.ts` shape — deterministic stdin ping → stdout response assertion.
- 11 live e2e smoke entries (one per server) in `docs/smoke/v1.8-mcp-e2e-<server>.md`.
- Any server that fails after 3-direction team → CHANGELOG entry "MCP server X removed in v1.8 because <reason>" + remove from `~/.copilot/mcp-config.json` template.

**What could fail**:
1. **`hermes-server.ts`** — known carve-out at `hermes-bridge.ts:223` (bare writeFileSync for log touch). If Copilot's MCP client doesn't tolerate detached child semantics, hermes is the canonical removal candidate.
2. **`python-repl-server.ts`** — depends on Python being on PATH; on dev box it is, but Copilot's spawn env may not inherit. Diagnostic: server spawns then immediately exits.
3. **`shared-memory-server.ts`** — name overlap risk with `project-memory`; if Copilot resolves duplicate "memory" tool names, calls go to the wrong server.
4. **Aggregate burn**: if 4+ servers each take 2h to debug, this story alone is a session.

### S-COST-GOV — US-05 cost-governor outer-loop equivalent

**Risk class**: CATASTROPHIC (v1.7 already owes this; cascade-blocks HUD column 6)

**Why this risk**: v1.7 ADR-deferred this; the user mandate "不准问题往后迁移" (no kick the can) was violated once with an explicit ADR — a second deferral degrades the ADR-poisoning chain. If we cannot solve US-05 in v1.8, HUD column 6 (cost/tokens) in v1.9 becomes empty → docs-site v2.0 has a feature-shaped gap → v2.0 ships visibly incomplete.

**The technical problem (already known)**: PostToolUse fires per-tool-call; mode.ts outer-loop only sees per-spawn (`mode.ts:376` spawnSyncCrossPlatform with `stdio: "inherit"` — we don't see tool calls). The v1.7 ADR named 3 options (A: per-spawn callback, B: new `cost-advice-emit` event, C: both).

**Recommended approach (drafted, see ADR section below)**: **Option C** — keep PostToolUse for accumulation (still works when Copilot eventually fires hooks on Windows correctly); add a per-spawn `mode.ts` post-spawn callback that emits HUD-shaped cost advice based on Copilot's stdout cost reporting (parse from `copilot -p` final output footer) OR from `.omcp/state/cost-<sessionId>.json` (which is what cost-governor writes to per-session at `src/hooks/cost-governor/index.ts:67`).

**Acceptance criteria**:
- Per-spawn cost summary written to `.omcp/state/cost-summary-<sessionId>.json` after each `copilot` exit in mode.ts.
- HUD column 6 reads from the per-spawn summary (not from PostToolUse stream).
- Backwards compat: if PostToolUse stream is also producing data (when Stop hook upstream fix lands), it merges cleanly.
- 1+ deterministic vitest + 1 live smoke artifact.

**What could fail**:
1. Copilot stdout doesn't expose cost in `-p` mode (only in interactive UI). Diagnostic: spawn `copilot -p "hello"` and inspect stderr/stdout for token counts. If absent → switch to estimating from prompt+completion bytes (rough, but ships).
2. Cost-governor hook writes a per-session file that mode.ts can't reliably locate (sessionId mismatch between hook ctx and mode.ts opts). Mitigation: thread sessionId from mode.ts → `OMCP_SESSION_ID` env var → cost-governor reads from env first.
3. Three-direction team needed if both A and B fail. ADR-deferral here would be **third strike** on this item — not acceptable. Hard rule: must solve in v1.8 or document explicit feature removal.

### S-MODE-AUTOPILOT — Live-verify `autopilot`

**Risk class**: MAJOR

**Why this risk**: autopilot's phase-transition spec (spec → plan → execute) is the most internally complex of the LOOPING_MODES. Its spawn semantics are shared with team/ultrawork; if autopilot's internal continuation is broken, team's worker spawn likely is too. Solving autopilot first unblocks team + ultrawork.

**Acceptance criteria**:
- Minimal PRD: "Write a Python function that returns the sum of two ints, with a vitest-style test."
- `omcp autopilot "<PRD>" --max-iterations 5` produces a working `add.py` + passing test in a tmpdir.
- Live smoke artifact: `docs/smoke/v1.8-autopilot.md`.

**What could fail**:
1. autopilot SKILL.md content stale vs. v1.7 mode.ts argv shape (missing --yolo handling).
2. Phase transition state file (`.omcp/state/autopilot-phase-<sessionId>.json`) not being written → autopilot loops on "spec" phase forever.
3. v1.7 stall detection in mode.ts only applies to `ralph`. If autopilot stalls, NO bail-out exists. Risk: a broken autopilot run burns Copilot rate-limit budget.

### S-MODE-TEAM — Live-verify `team` (4-worker shard-merge)

**Risk class**: MAJOR

**Why this risk**: L2.5b fixing-phase + ack protocol is untested; pidfile invariant 9 stress test (4 concurrent pidfiles at `.omcp/state/team/<sessionId>/worker-K.pid` per `team.ts:89`); shard-merge conflicts are filesystem-level (the worst kind to debug).

**Acceptance criteria**:
- 4-worker team on a tasklist of 4 independent files (no overlap) merges cleanly.
- 4-worker team on a tasklist of 4 overlapping files triggers fixing-phase + ack protocol correctly.
- All 4 pidfiles cleaned up post-completion (Invariant 9).
- Live smoke artifact + 1 deterministic vitest verifying ack protocol state machine.

**What could fail**:
1. Pidfile race: 2 workers writing to the same `.omcp/state/team/<sessionId>/` dir at the same ms. Mitigation: per-worker dir already exists; verify Invariant 9 holds.
2. Fixing-phase detection logic depends on a Stop-hook-style signal that doesn't fire on Windows (the very upstream bug v1.6 outer-loop works around). Diagnostic: team mode may have NEVER worked on Windows for the same reason ralph didn't pre-v1.6.

## ADR candidates (drafted)

### ADR-C1 (mandatory for v1.8): US-05 cost-governor outer-loop equivalent

- **Decision**: Implement **Option C** — keep PostToolUse hook for accumulation; add a per-spawn post-spawn callback in `mode.ts` that writes `.omcp/state/cost-summary-<sessionId>.json` based on whichever of (a) Copilot stdout cost reporting in `-p` mode or (b) byte-count estimation is available.
- **Drivers**: HUD column 6 dependency (v1.9); v1.7 already deferred once; "no kick the can" hard rule.
- **Alternatives considered**:
  - A (per-spawn callback only): loses per-tool-call granularity forever, even after upstream Stop-hook fix.
  - B (new `cost-advice-emit` event only): adds a new event to Invariant 4's valid set; one more thing to keep in sync.
  - C (both): best of both; pays small complexity cost.
- **Why C chosen**: forward-compatible with upstream fix; ships value now.
- **Consequences**: One new state file shape; one new env var (`OMCP_SESSION_ID` threaded from mode.ts).
- **Follow-ups**: HUD column 6 reads from the per-spawn summary in v1.9.
