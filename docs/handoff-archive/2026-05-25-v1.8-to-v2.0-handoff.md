## v1.8 work plan (operational, not theoretical)

Per roadmap, v1.8 has 4 tiers. **Suggested execution order** (highest-
ROI risk first):

### Phase 1 (one session): Tier 3 MCP server e2e — highest risk

The 11 MCP servers have **never been live-verified end-to-end**. v1.5
fixed plugin-install deps; that confirms they CAN start. v1.8 must
prove each one actually responds to MCP protocol from a real autopilot
session.

Per MCP server (state / notepad / trace / project-memory / loop / code-
intel / hermes / wiki / python-repl / shared-memory / server-runtime entry):
1. **Deterministic harness**: spawn the server's CLI entry → ping
   over stdin → assert valid MCP JSON-RPC response. 1 vitest per server.
2. **Live e2e**: from a real `omcp autopilot` smoke against a tiny task
   that requires that MCP server, verify the tool call shape + return.

Failure → multi-direction team OR explicit CHANGELOG-tagged removal.
**Anti-pattern**: NEVER silently leave a broken MCP server in default
list and push verification to v1.9.

### Phase 2 (one session): Tier 1 LOOPING_MODES live verify

Per mode (ralph already done; autopilot / team / ultrawork / ultraqa /
sciomc remaining):
1. Design minimal PRD that exercises the mode's distinctive feature
2. Run `omcp <mode> "<task>" --max-iterations N` in a tmp dir
3. Verify expected end-state (PRD passes / generated artifacts /
   ralph-state shape)

Per the "tag gate = live evidence" principle: deterministic vitest
mock-spawn is NOT sufficient for tag gate — must produce real Copilot
ralph-state.json / output artifacts to count.

Failure → multi-direction team (CLI argv gap? SKILL.md content wrong?
agent catalog wrong?) until pass or CHANGELOG-tagged removal.

### Phase 3 (half session): Tier 4 + Tier 2

Tier 4 (19 agent QA matrix): diff each agent against omc, verify
model_family pair, tool surface (no Claude-only names), each has a
mock-spawn deterministic test.

Tier 2 (one-shot modes CLI smoke): ask / ralplan / plan / deep-interview
/ deep-dive / external-context / note / learner / cancel / hud /
visual-verdict — each gets CLI argv smoke verifying skill invocation
succeeds (lower bar than LOOPING_MODES because no iteration semantics).

### Phase 4 (in same session as cut): US-05 cost-governor outer-loop

The v1.7 ADR-deferral. Design challenge: PostToolUse fires per-tool-call;
mode.ts outer-loop only sees per-spawn. Options:
- (A) Move cost-governor to a per-iteration post-spawn callback in mode.ts;
  lose per-tool-call granularity but gain reliability
- (B) Add a new "cost-advice-emit" hook event that mode.ts fires post-
  spawn; cost-governor subscribes to that instead of PostToolUse
- (C) Both: keep PostToolUse for tool-level metrics, add post-spawn for
  per-iteration cost advice

Recommend multi-direction team to pick approach.

## What NOT to do (lessons from v1.3 → v1.7 sessions)

1. **Don't tag before live smoke evidence.** v1.3-v1.5 shipped tags
   while user-facing iteration-advance bug persisted because deterministic
   tests didn't catch it. v1.6 established the gate; v1.7 enforced it.
   Don't lower it.

2. **Don't push carry-forward without an ADR.** v1.5 silently deferred
   DEP0190 → bit us until v1.7. If you can't solve in current version,
   write an explicit ADR with technical reason; don't just say "v1.X".
   v1.7's US-05 ADR-deferral is the pattern; emulate it.

3. **Don't try single-agent on hard problems.** DEP0190 single-agent
   attempt in v1.5 failed; multi-direction team (3 parallel agents)
   in v1.7 converged on same solution. The "3+ parallel directions"
   principle works — use it when single-agent stalls >30 min.

4. **Don't trust hook reminders saying "Edit operation failed"** when
   tool result says "updated successfully". The hook reminders are
   noisy across many sessions; tool result is authoritative. If a real
   edit fails, the next operation (read / tsc / test) will surface it.

5. **Don't `cd` to tmp dir then run follow-up without `cd` back to
   repo root.** Background bash commands lose cwd. Use absolute paths
   or chain with `&&`.

6. **Don't file upstream issues from runjiashi_microsoft EMU account** —
   it's blocked from public github/copilot-cli. User handles upstream
   filing personally via personal account.

7. **Don't push tags before v2.0.** Local-tag only. v2.0 is the first
   push + publish.

8. **Don't expand v1.8 scope mid-version.** If a v1.8 finding suggests
   a v1.9-shaped feature, write an ADR documenting it as v1.9 scope
   and move on. Within-version findings stay in-version; new shape
   findings flow forward with ADR.
