## Context

The pre-RG live-multi-worker harness shipped at v2.2.0 has a real,
demonstrable correctness gap: two workers writing the same shard file
will clobber each other today, and `team-verify` has no way to surface
the collision. This is not a hypothesized failure mode — it is
observable in the existing harness with a 2-worker test that writes
non-overlapping line ranges to the same file. The last writer wins;
the loser's work is silently destroyed; merge-verification sees
"clean tree" because the file exists and parses.

The RALPLAN-DR plan §3 frames this as **gap 3 of 6** in Robin's fork
comparison: multi-worker file collision detection. Of the six gaps,
RG-03 is the **only** one that addresses a correctness loss reproducible
today in our existing test harness (the others — receipt protocol,
active push, event log — are insurance for failure modes that are
real but not currently surfaced). The S2 parity-theater steelman in
the plan explicitly distinguishes RG-03 from the rest: "RG-03 has
demonstrable correctness value (two workers writing same shard
clobber today + verify can't surface). RG-01/RG-02 are cheap insurance
on top of that demonstrable need."

Robin Norberg's fork addresses the same need through his MCP bridge's
in-process write coordination. The bridge serializes writes per shard
through a runtime lock table; collisions never reach the filesystem.
Adopting that pattern would invert our architecture (Option A from the
plan, rejected). Hybrid options that signal collision **after** the
fact (post-merge `git diff`, mtime comparison, etc.) suffer from race
windows where the loser cannot prove it lost.

The conflict mailbox primitive shipped at commit `5a58cd5` chose a
third path: a **verb-layer signal** where workers self-report observed
collisions via a new `team-conflict-write` verb, the records live on
per-shard JSONL streams, `team-verify` pre-flight-scans those streams,
and human resolution flows through a `team-conflict-ack` companion
verb. The conflict mailbox is a coordination primitive, not a
collision detector — workers must observe their own collisions, then
report them. This trade-off is documented in detail under Option E
below.

A retention concern surfaced in iter-1 critic review (C3 / PM-E): a
busy session repeatedly colliding on the same shard would grow the
conflict file without bound, causing `team-verify` linear-scan latency
to degrade. The mitigation chosen was the 1MB rotation pattern
analogous to RG-04a, with the rotation happening INSIDE the
per-stream lockfile per the architect A2 rotation-lock contract.
Ack-deletable record semantics (sibling `.acked.jsonl` file) allow
operators to mark conflicts resolved so the default `team-conflict-read`
view stays tidy.

## Decision

**Ship three new stateless verbs (`team-conflict-write`,
`team-conflict-read`, `team-conflict-ack`) operating on per-shard
JSONL streams at `.omcp/state/team/<sid>/conflicts/<shard>.jsonl`,
with 1MB rotation inside the per-stream lockfile and a sibling
`<shard>.acked.jsonl` file for resolved-conflict filtering. Extend
`team-verify` with a pre-flight conflict scan that forces
non-zero exit until unresolved conflicts are acked.**

Concrete shape:

1. **`team-conflict-write <sid> <shard> <worker-id> <attempted-op>
   [--rationale "..."]`** — appends one record to
   `.omcp/state/team/<sid>/conflicts/<shard>.jsonl`. Record shape:
   `{conflict_id, ts, shard, worker_id, attempted_op, rationale,
   producer_fork}`. `conflict_id` is a UUIDv4 generated by the verb;
   returned to the caller so the human resolver can ack by id.

2. **`team-conflict-read <sid> <shard> [--include-acked] [--json]
   [--exit-nonzero-if-unresolved]`** — reads the per-shard conflict
   stream and emits records to stdout. Default view filters out
   conflict_ids that appear in the sibling `<shard>.acked.jsonl`.
   The `--exit-nonzero-if-unresolved` flag composes with
   `team-verify`'s pre-flight scan.

3. **`team-conflict-ack <sid> <shard> <conflict-id>`** — appends an
   ack record to `<shard>.acked.jsonl`. `conflict-id` MUST be a
   UUIDv4 (the value returned by an earlier `team-conflict-write`).
   Acks are immutable append-only; an accidentally-acked conflict
   cannot be un-acked through the verb layer (use direct file edit
   as the operator escape hatch).

4. **1MB rotation inside per-stream lock (PM-E + A2)** — when a write
   acquires the per-stream lock and observes the stream file
   exceeding 1MB, it renames `<shard>.jsonl` → `<shard>.jsonl.1`
   inside the lock (overwriting any existing `.1`). Only one process
   can hold the lock at a time, so the rotation race is eliminated.
   The new write then appends to a fresh `<shard>.jsonl`.

5. **`team-verify` pre-flight scan** — before declaring shard merges
   clean, `team-verify` calls `runTeamConflictRead` per shard.
   Unresolved conflicts force `ok=false`, exit non-zero, and surface
   in the verify report's new `unresolved_conflicts` field. After
   the human resolves the collision (manual file edit) and calls
   `team-conflict-ack` for each id, verify exits 0.

6. **Cross-fork attribution** — every conflict record carries the
   `producer_fork: "omcp-r2"` field per the ADR-RG-01 contract.
   Records lacking the field surface in `team-conflict-read`'s
   `parseErrors` output (treated as malformed; not silently
   matched).

## Drivers (top 3)

1. **Demonstrable correctness gap (the only one in EB-RG).** Two
   workers writing the same shard today clobber each other, and
   `team-verify` cannot surface the collision. This is reproducible
   in the existing live-multi-worker harness without any new
   instrumentation. Driver pinned the choice to ship this primitive
   even when RG-01/RG-02 might have been deferred — the S2
   parity-theater rebuttal in the plan §3 names RG-03 as the
   correctness anchor of the entire EB-RG arc.

2. **Unbounded growth on busy sessions (PM-E / C3).** Append-only
   JSONL with no rotation grows linearly with conflict count.
   `team-verify` linear-scan latency degrades proportionally. An
   integration test on a 10,000-conflict synthetic load would
   surface degradation in practice. Driver pinned the 1MB rotation
   inside per-stream lock + ack-deletable record semantics as the
   retention strategy.

3. **Stateless-verb DNA (Principle 1).** The Robin-bridge alternative
   serializes writes through an in-process lock table that lives as
   long as the session. Our primitive is three short-lived verbs
   that observe-and-report rather than coordinate. Workers self-
   report collisions; verify gates on unresolved records; humans
   resolve. No daemon, no supervisor, no in-process state.

## Alternatives Considered

### Option A — Pessimistic locking on shard files (rejected)

- **Scope**: before a worker writes a shard file, acquire a
  per-shard write lock. Other workers block on the lock until the
  first releases. Collision becomes impossible because writes are
  serialized.
- **Pros**: collisions never reach the filesystem; no need for a
  conflict mailbox or verify pre-flight scan; semantically clean.
- **Cons**: deadlock risk on cyclic dependencies between shards.
  Worker A holds shard X waiting for shard Y; worker B holds shard
  Y waiting for shard X. Robin's bridge solves this with a runtime
  lock manager that detects cycles and breaks them — a primitive we
  do not have at the verb layer.
- **Rejection rationale**: deadlock detection requires a long-running
  arbiter; violates Principle 1. The verb-layer alternative
  (observe-and-report) accepts collisions and reports them rather
  than preventing them.

### Option B — Hash-mismatch detection at write time (rejected)

- **Scope**: every shard write reads the file, hashes the contents,
  performs the write, re-hashes, and compares. Mismatch (because
  another writer interleaved) triggers a conflict-write call.
- **Pros**: detects collisions at write time; no need for workers
  to "observe" their own collisions — the verb does it for them.
- **Cons**: extra IO round-trip per shard write (read-then-hash on
  the write path). Iter-1 architect review §5 RG-03 flagged this as
  "acceptable cost for rare conflict events" but the implementation
  team rejected it during commit prep — see "Rejected" line in the
  commit body of `5a58cd5`.
- **Rejection rationale (from commit `5a58cd5`)**: "hash-mismatch
  via read+hash+write+re-hash (extra IO round trip; replaced with
  'verb-layer signal on collision': workers emit team-conflict-write
  when collision observed; team-verify gates merge)." The
  observe-and-report pattern is simpler at the verb layer and
  shifts the detection burden to the worker (who is best positioned
  to know what it was trying to write).

### Option C — mtime-based detection (rejected)

- **Scope**: workers record the shard file's mtime before write;
  re-check after; mismatch indicates another writer landed in
  between.
- **Pros**: cheap; no IO beyond a `stat` call.
- **Cons**: false positives under filesystem clock skew (NTFS
  mtime quantum is 15.625ms per `ADR-omcp-eb-05-heartbeat-freshness`
  Driver 2). Two writers landing in the same quantum see identical
  mtime despite the collision. The whole point of conflict
  detection is to avoid silent data loss; mtime-based detection
  produces false confidence.
- **Rejection rationale**: same Driver 2 that pinned heartbeat-ts-
  primary over mtime in EB-05 applies here. Hash-vs-mtime ladder is
  already documented in ADR-RG-03 §5 of the plan; mtime loses.

### Option D — Verify-time detection only (rejected)

- **Scope**: skip the write-time signal entirely. At
  `team-verify`-time, recompute file hashes against a baseline and
  flag any divergence as a potential collision.
- **Pros**: no per-write overhead; no new write-time verbs.
- **Cons**: by verify-time the collision has already happened. The
  losing writer has no record of what it tried to write; the human
  resolver has no `attempted_op` or `rationale` to consult. Verify
  can flag the file as "changed since last verify" but cannot
  attribute the change to a colliding worker or produce a
  resolution workflow.
- **Rejection rationale**: late detection is worse than no detection
  for the workflow that needs the data — the human needs the
  rationale to choose which version to keep.

### Option E — Append-only with file rebase / merge semantics (rejected)

- **Scope**: treat each shard as a sequence of append-only operations.
  Collisions trigger an automatic rebase or 3-way merge analogous to
  git's behavior on text files.
- **Pros**: collisions become recoverable without human intervention
  for trivial cases.
- **Cons**: architectural overreach. Implementing rebase semantics at
  the verb layer requires a diff/merge library (new dep, violates
  Driver 3) or a hand-rolled merge primitive (significant LoC).
  Robin's fork does not do this either; he serializes through the
  bridge instead.
- **Rejection rationale**: scope creep. RG-03 is a coordination
  primitive, not a merge tool. Merge tools belong in `team-verify`
  proper or as a separate verb.

## Why Verb-Layer Signal Chosen

- **Demonstrable correctness without architectural overreach.** The
  primitive surfaces collisions without preventing them. Workers
  detect their own collisions (they have the most context); the
  verb persists the conflict record; verify gates the merge until
  human resolution.
- **Stateless-verb DNA preserved.** Three short-lived verbs +
  pre-flight scan extension to existing `team-verify`. No daemon,
  no in-process lock table, no kernel handles.
- **Reuses per-stream lockfile primitive from RG-04a.** The
  `acquirePerStreamLock` helper extracted in `src/runtime/per-
  stream-lock.ts` (RG-04a) is shared with conflict streams. Both
  stories rely on the same A2 rotation-lock contract.
- **Rotation cost is bounded.** 1MB rotation analogous to RG-04a.
  One conflict record is ~150-200 bytes (`conflict_id` UUID, ts,
  shard path, worker id, attempted op, rationale string,
  producer_fork). 1MB ≈ 5,000-7,000 conflicts before rotation —
  more than any realistic session generates.
- **Ack-deletable semantics scale to operator workflow.** Resolved
  conflicts disappear from the default `team-conflict-read` view
  without losing the audit trail (the original record remains in
  `<shard>.jsonl`; the ack record is in `<shard>.acked.jsonl`).
  Operators inspect the full history by passing `--include-acked`.

## Consequences

### Visible to users

- Three new verbs at the CLI:
  - `omcp team-conflict-write <sid> <shard> <worker-id>
    <attempted-op> [--rationale "..."]`
  - `omcp team-conflict-read <sid> <shard> [--include-acked]
    [--json] [--exit-nonzero-if-unresolved]`
  - `omcp team-conflict-ack <sid> <shard> <conflict-id>`
- `team-verify` now pre-flight-scans conflict streams. A session
  with unresolved conflicts will fail verify until each conflict is
  acked. This is intentional friction — verify must not declare
  "clean" when collision records exist.
- The verify report gains a new `unresolved_conflicts` field
  enumerating shard + conflict_id pairs.

### Visible to operators

- Per-shard conflict files at
  `.omcp/state/team/<sid>/conflicts/<shard>.jsonl` and sibling
  `<shard>.acked.jsonl` files (one ack record per resolved
  conflict). Files grow until rotated at 1MB. Verify-time
  pre-flight scan adds ~10ms per shard for stat + read of small
  files (rough estimate; not benchmarked).
- Conflict files grow until acked. A busy session that does not
  resolve conflicts will see verify continue to fail. This is the
  designed behavior — surfacing the problem rather than hiding it.
- Rotation events do not emit log lines by default; RG-04b
  instrument-conflict commit (if/when it lands) will wire
  `team-event-append` into the conflict verbs.

### Visible to maintainers

- Workers must self-report collisions. The conflict mailbox does
  NOT auto-detect collisions; it persists what workers tell it.
  Worker SKILL discipline applies: when a worker observes
  (a) a shard file has changed since it last read it, OR (b) its
  write would clobber another worker's recent write, it must call
  `team-conflict-write` rather than proceeding silently. Workers
  shipping with pre-v2.3 SKILL.md continue to clobber silently;
  RG-03 only helps when paired with the updated SKILL.
- `src/runtime/per-stream-lock.ts` is shared between RG-03 and
  RG-04a. Bug fixes in the lockfile primitive benefit both
  consumers but also require regression testing both call sites.
- The `team-verify.ts` modification is limited to pre-flight scan
  insertion. Existing verify logic is untouched; the pre-flight
  scan never replaces verify, only gates it.

### Visible in the test pyramid

- 36 new vitest cases in `src/__tests__/team-conflict.test.ts`
  covering: write/read/ack happy paths, two-worker collision,
  1MB rotation under per-stream lock, ack-deletable filter,
  argv validation, CLI wrappers, producer_fork stamps.
- 51 existing `team-verify` tests continue to pass — regression-
  clean across the verify pre-flight scan insertion.
- Smoke row `conflict-detect` is DEFERRED to RG-05 per architect
  A5 distributed-smoke contract (consistent with RG-01 + RG-02).

## Follow-ups

1. Conflict rotation policy refinement if unacked conflicts
   accumulate at the rotation boundary. Currently the rotation is
   "overwrite the existing `.1`" — if `.jsonl` is 1MB AND `.1` is
   also 1MB of unacked records, the old `.1` records are lost on
   rotation. Acceptable for v2.3 (1MB is far above typical conflict
   density) but worth flagging for v3.x.
2. Surface conflict-rate metric in `team-event-health-check`
   observability dashboard (RG-05). Rapid conflict accumulation
   indicates workflow misconfiguration (e.g. two workers assigned
   the same shard).
3. Worker SKILL update — the SKILL.md instruction to call
   `team-conflict-write` on observed collision is a v2.3 ship-with
   item. Pre-v2.3 workers are silent on collision; users mixing
   versions get partial coverage.
4. RG-04b instrument-conflict commit will wire `team-event-append`
   into the three conflict verbs. The instrumentation lands as a
   separate commit so reverting RG-03 also reverts its instrument
   patch.
5. Document the verb-layer signal pattern as the v2.x conflict
   detection contract. Any future "conflict detection" verb (e.g. a
   write-time wrapper that auto-emits `team-conflict-write` on
   observed mtime change) should consume this primitive rather
   than ship its own JSONL stream.
