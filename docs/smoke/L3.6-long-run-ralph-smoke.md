### 🔍 Why the housekeeping gaps?

Hypothesis: Copilot in `--autopilot` mode completed all 10 stories in a SINGLE
Copilot turn (no continuation needed because each story was trivial enough to
fit in one turn). This means:

- Only 1 Stop event fired (at session-end), not 10 (one per story)
- The persistent-mode hook's iteration increment + allComplete clear ran ONCE
  at session-end
- BUT ralph-state.json still shows `iteration:1, active:true` — suggesting
  either (a) the Stop hook DID run but state-clear didn't fire for some
  reason, OR (b) the Stop hook code-path didn't get the right context to
  detect allComplete

Possible root causes (for v1.4 RCA):
1. `--autopilot` mode might dispatch Stop events differently than non-autopilot
2. `incrementRalphIteration` race: ralph-state read at hook-start might already
   have the post-spawn snapshot; increment writes back original value
3. `getPrdCompletionStatus` race: PRD might still show some `passes:false` at
   Stop-time (Copilot writes prd.json AFTER its final reply)
4. `clearRalphState` conditional logic (L3.3) might not be hitting the
   intended branch under this particular spawn/exit timing

## What this smoke validates

✓ ralph's core ability to drive a real Copilot session through 10 PRD stories
✓ Each story's acceptance criteria evaluation (Copilot self-verified)
✓ exit-code propagation
✓ File-writing tool integration (no spawn failure)
✓ The upstream Copilot CLI hook-dispatch bug (still present, same scale as v1.1)

## What this smoke does NOT validate (gaps + follow-up work)

✗ L3.1 progress.txt cap under realistic load (file never created)
✗ L3.2 compaction re-arm at iter 5 boundary (utilization too low to fire)
✗ L3.3 conditional clearRalphState (didn't fire — needs RCA)
✗ L3.4 stale mode-state resume (not exercised — no stale state to start)
✗ L1.3 Stop-side compaction advise actual delivery (utilization too low; needs
   a 30+ iter smoke with larger payloads to cross 80% threshold)

## v1.4 follow-up RCA tasks

1. **ralph-state housekeeping under autopilot**: investigate why `iteration`
   doesn't advance + `clearRalphState` doesn't fire when Copilot completes
   multiple stories in a single autopilot turn. Likely an interaction between
   `incrementRalphIteration` timing and the Stop-event dispatch under
   `--autopilot`.
2. **High-load compaction smoke**: re-run with a PRD that has 30+ stories
   requiring real file edits (not trivial writes) to force context utilization
   above 50%, then verify L1.3 advise actually delivers.
3. **progress.txt visibility**: surface a directive in the ralph SKILL telling
   Copilot to call `omcp state write progress -- "<note>"` after each story so
   L3.1 cap can be exercised under realistic load.

## Conclusion

orchestrator-v1's core promise — "PRD-driven ralph loop completes a 10-story
workload end-to-end against real Copilot" — **is validated**. ralph cuts
the work, Copilot executes, PRD stories mark passes:true, exit 0. The
housekeeping layer (state cleanup + iteration counter) has gaps under
`--autopilot` that don't affect functional behavior but do reduce observability
of long runs. v1.3.0 ships with this caveat documented; v1.4 picks up the RCA.

The L1.3 + L3.x infrastructure landed in v1.2.0 + extended in v1.3.0 is
**code-complete + unit-tested**; v1.4 will close the live-validation gap once
we have a smoke that genuinely stresses the long-running paths.
