## Why "ralph-state cleared" does NOT prove Stop hook ran

The mode.ts housekeeping fix (commit `3175be5`) is a fallback that runs in
the omcp process after the spawned `copilot` exits:

```ts
// mode.ts:298-318 (clean exit branch)
const postRunPrd = getPrdCompletionStatus();
const shouldClear = postRunPrd.allComplete || preSpawnRalphSnapshot?.architectApproved === true;
if (!shouldClear && preSpawnRalphSnapshot) {
  writeRalphState(preSpawnRalphSnapshot);  // preserve for resume
}
// else: clearModeState above already removed the file — done.
```

This branch fires REGARDLESS of whether Stop hook handlers ran. As long
as the PRD ends in an all-`passes:true` state (which Copilot accomplished
via tool calls regardless of hook delivery), `shouldClear` resolves true
and state stays cleared.

So the "state cleared" outcome on this smoke is **uniquely consistent
with EITHER**:
- (A) Stop hooks ran + cleared state themselves (the design intent), OR
- (B) Stop hooks all failed + mode.ts post-spawn cleanup handled it

The log evidence below shows reality is (B).

Sample Stop-event handler failure (representative of all 3 Stop events
fired this smoke):

```
2026-05-24T10:51:43.223Z [ERROR] Hook execution failed: HookExitCodeError: Hook command failed with code 1
Stderr: [stdin]:1
{"hook_event_name":"Stop","session_id":"51fd70c0-...","timestamp":"2026-05-24T10:51:41.694Z","cwd":"C:\\Users\\runjiashi\\Temp\\omcp-v14-smoke","transcript_path":"C:\\Users\\runjiashi\\.copilot\\session-state\\51fd70c0-...\\events.jsonl","stop_reason":"end_turn"}
                  ^
Expected ';', '}' or <eof>

SyntaxError: Unexpected token ':'
    at makeContextifyScript (node:internal/vm:194:14)
```

**Diagnosis**: this is the **upstream Copilot Windows pwsh dispatch
bug** previously documented in
`docs/upstream-reports/copilot-cli-hook-eval-stdin.md`. Copilot
1.0.53-2 dispatches the hook command via `pwsh.exe`. On Windows, the
script-path argument is being dropped from the invocation. Node
receives stdin (the payload JSON) but no script file argument, enters
`node:internal/main/eval_stdin` mode, and tries to parse the JSON as
TypeScript — failing at the first colon (`{"hook_event_name":`).

This means **omcp's hook handler code never ran** for any of the 3
Stop events in this smoke. Therefore:
- `incrementRalphIteration` did not fire
- `clearRalphState` from the hook side did not fire
- Bail-out guards (isContextLimitStop / isRateLimitStop) did not run

The cleanup we observed (state file gone) was 100% from mode.ts
fallback, not from the Stop hook pipeline.
